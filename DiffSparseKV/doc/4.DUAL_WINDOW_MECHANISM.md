## 核心概念

### 🎯 双窗口机制的目的

在 Decoding 阶段，为了准确判断 token 的重要性，需要收集足够的 attention 信息。双窗口机制确保：
- **每个 token 在被压缩前，都被足够多的后续 tokens "观察"过**
- **避免过早压缩可能重要的 token**

### 📐 基本设计

```
Window A (观察窗口): 128 tokens
- 等待被压缩的 tokens
- 正在收集 attention 信息

Window B (保护窗口): 128 tokens  
- 新生成的 tokens
- 暂时不压缩，保持完整
- 用于"观察" Window A 的 tokens
```

## 为什么需要双窗口

### ❌ 单窗口的问题

假设只有一个窗口（128 tokens）：

```
窗口: [Token 1, Token 2, ..., Token 127, Token 128]
                                          ↑
                                    最后一个 token
```

当要压缩这个窗口时：
- **Token 1-127**：已经被后续的 tokens 关注过，有足够的 attention 信息 ✅
- **Token 128**：刚加入窗口，还没有被后续 tokens 关注过 ❌
  - 无法准确判断它的重要性
  - 可能会错误地压缩掉重要的 token

### ✅ 双窗口的解决方案

```
Window A: [Token 1 ... Token 128]  ← 等待压缩
Window B: [Token 129 ... Token 256] ← 新生成的

当 Window B 满了（128 tokens）：
- Window A 的所有 tokens（包括 Token 128）都被 Window B 的 128 个 tokens "观察"过
- 现在可以安全地压缩 Window A
- Window B → Window A（滑动）
- 清空 Window B，继续生成
```

## 窗口的定义

### 🔧 物理结构 vs 逻辑划分

**重要**：Window A 和 Window B 只是**逻辑上的划分**，在 KV Cache 中是**连续存储**的。

```python
# KV Cache 的物理结构（连续存储）
KV_cache = [Token_1, Token_2, Token_3, ..., Token_N]

# Window A 和 Window B 通过索引标识
Window_A_indices = [start_A, start_A+1, ..., start_A+127]
Window_B_indices = [start_B, start_B+1, ..., start_B+127]
```

### 📍 窗口的初始化

#### 场景 1：Prefill 阶段有足够的 tokens（≥ 128）

```python
Prefill 输入: [Token_1, Token_2, ..., Token_1000]

Prefill 处理：
├─ Token_1 到 Token_872:  应用 DiffSparseKV 稀疏化
└─ Token_873 到 Token_1000: 保护，不稀疏（最后 128 个）
                            ↑ residual_length = 128

结果：
[稀疏化的 tokens] [Token_873 ... Token_1000]
                  ↑ 128 个完整的 tokens
                  ↑ 这就是初始的 Window B
```

#### 场景 2：Prefill 阶段 tokens 不足 128 个

```python
Prefill 输入: [Token_1, Token_2, ..., Token_50]

Prefill 处理：
└─ 全部保护，不稀疏（因为不足 128 个）

进入 Decoding，生成新 tokens：
[Token_1 ... Token_50] [Token_51 ... Token_128]
↑ Prefill (50)        ↑ Decoding (78)
↑ 合起来 128 个 tokens = 初始 Window B

继续生成到 256 个 tokens 时：
[Token_1 ... Token_128] [Token_129 ... Token_256]
↑ Window A              ↑ Window B
```

## Attention 累积机制

### 🔄 Decoding 阶段的 Attention 计算

在 Decoding 阶段，每生成一个新 token：

```python
# 第 k 步生成
new_query = Q_new  # [B, H, 1, D] - 只有一个 query
all_keys = [... 所有之前的 tokens ...]  # [B, H, seq_len, D]

# 计算 attention
attn_weights = softmax(Q_new @ all_keys.T / sqrt(d))  # [B, H, 1, seq_len]

# 这个 query 会关注所有之前的 keys，包括：
# 1. 稀疏化的历史 tokens
# 2. Window A 的 128 个 tokens
# 3. Window B 中已经生成的 tokens
```

### 📊 累积过程详解

假设当前状态：

```
[... 稀疏化 ...] [Window A: Token_873 ... Token_1000] [Window B: 空]
                 ↑ 128 个 tokens，等待收集 attention
```

开始生成 Window B 的 tokens：

```python
# ===== 生成第 1 个新 token (Token_1001) =====
Query: Token_1001
Keys:  [... 稀疏化 ...] + [Token_873 ... Token_1000]

Attention: Token_1001 关注所有之前的 keys
累积 Window A 的 attention:
- Token_873:  +1
- Token_874:  +1
- ...
- Token_1000: +1

# ===== 生成第 2 个新 token (Token_1002) =====
Query: Token_1002
Keys:  [... 稀疏化 ...] + [Token_873 ... Token_1000] + [Token_1001]

Attention: Token_1002 关注所有之前的 keys
累积 Window A 的 attention:
- Token_873:  +1 (总共 2)
- Token_874:  +1 (总共 2)
- ...
- Token_1000: +1 (总共 2)

# ===== 继续生成... =====

# ===== 生成第 128 个新 token (Token_1128) =====
Query: Token_1128
Keys:  [... 稀疏化 ...] + [Token_873 ... Token_1000] + [Token_1001 ... Token_1127]

Attention: Token_1128 关注所有之前的 keys
累积 Window A 的 attention:
- Token_873:  +1 (总共 128)
- Token_874:  +1 (总共 128)
- ...
- Token_1000: +1 (总共 128)

此时 Window B 满了，触发压缩！
```

### 💾 累积器的实现

```python
# 初始化累积器
attention_accumulator = torch.zeros([B, H, 128])  # 为 Window A 的 128 个 tokens

# 生成 Window B 的每个 token
for k in range(128):
    # 生成新 token
    new_query = generate_query()  # [B, H, 1, D]
    all_keys = get_all_keys()     # [B, H, seq_len, D]
    
    # 计算 attention
    attn_weights = compute_attention(new_query, all_keys)  # [B, H, 1, seq_len]
    
    # 提取 Window A 的 attention 并累积
    window_A_attention = attn_weights[:, :, :, Window_A_indices]  # [B, H, 1, 128]
    attention_accumulator += window_A_attention.squeeze(2)  # [B, H, 128]

# 当 Window B 满了，attention_accumulator 包含了完整的累积信息
```

## 归一化计算

### 🔑 关键洞察：不同 token 的累积次数不同

Window A 的 tokens 不仅被 Window B 关注，它们在生成时也**相互关注**过！

```python
Window A = [Token_873, Token_874, ..., Token_999, Token_1000]
           ↑ 第 0 个                              ↑ 第 127 个

这些 tokens 在生成时的相互关注（Causal Attention）：
- Token_873:  被 Token_874, 875, ..., 1000 关注 = 127 次
- Token_874:  被 Token_875, 876, ..., 1000 关注 = 126 次
- Token_875:  被 Token_876, 877, ..., 1000 关注 = 125 次
- ...
- Token_999:  被 Token_1000 关注 = 1 次
- Token_1000: 没有被 Window A 内的 tokens 关注 = 0 次
```

现在生成 Window B，又被关注：

```python
Window B 生成期间：
- Token_873:  被 Token_1001, 1002, ..., 1128 关注 = 128 次
- Token_874:  被 Token_1001, 1002, ..., 1128 关注 = 128 次
- ...
- Token_1000: 被 Token_1001, 1002, ..., 1128 关注 = 128 次
```

### 📐 总累积次数

```python
Window A 的第 i 个 token (i = 0 to 127):

历史累积（Window A 内部）: 127 - i 次
Window B 累积: 128 次
总累积: (127 - i) + 128 = 255 - i 次

具体：
- Token[0]:   127 + 128 = 255 次
- Token[1]:   126 + 128 = 254 次
- Token[2]:   125 + 128 = 253 次
- Token[3]:   124 + 128 = 252 次
- ...
- Token[126]: 1 + 128 = 129 次
- Token[127]: 0 + 128 = 128 次
```

### ✅ 归一化公式

```python
for i in range(128):
    total_count = 255 - i
    attention_avg[i] = attention_accumulator[i] / total_count

# 或者更明确地：
attention_avg[0] = attention_accumulator[0] / 255
attention_avg[1] = attention_accumulator[1] / 254
attention_avg[2] = attention_accumulator[2] / 253
...
attention_avg[126] = attention_accumulator[126] / 129
attention_avg[127] = attention_accumulator[127] / 128
```

### 🎯 为什么每轮都一样？

**重要发现**：无论是第几轮压缩，累积次数的模式都是一样的！

```python
第一轮：
Window A = Prefill 的最后 128 个 tokens
- 它们在 Prefill 时相互关注过
- 累积次数：255, 254, 253, ..., 129, 128

第二轮：
Window A = 上一轮的 Window B
- 它们在生成时相互关注过
- 累积次数：255, 254, 253, ..., 129, 128

第三轮：
Window A = 上一轮的 Window B
- 它们在生成时相互关注过
- 累积次数：255, 254, 253, ..., 129, 128

...
```

**原因**：
1. Window A 总是 128 个连续的 tokens
2. 这 128 个 tokens 在生成时就相互关注了（causal attention）
3. Window B 满了才触发压缩，所以 Window A 总是被 128 个新 tokens 关注

## 完整工作流程

### 🔄 第一轮压缩

```python
# ===== Prefill 结束 =====
KV_cache = [... 稀疏化的 tokens ...] [Token_873 ... Token_1000]
                                      ↑ 128 个完整 tokens (初始 Window B)

# ===== 进入 Decoding，生成 Window B =====
attention_accumulator = zeros([B, H, 128])

for k in range(128):  # 生成 128 个新 tokens
    # 生成 Token_{1000+k+1}
    query = generate_query()
    keys = get_all_keys()  # 包括 Window A (Token_873...Token_1000)
    
    # 计算 attention
    attn = compute_attention(query, keys)
    
    # 累积 Window A 的 attention
    attention_accumulator += attn[:, :, Window_A_indices]

# ===== Window B 满了，触发压缩 =====
# 1. 归一化 attention
for i in range(128):
    attention_avg[i] = attention_accumulator[i] / (255 - i)

# 2. 计算 importance
importance = compute_diffkv_importance(attention_avg)

# 3. 使用 prefill 的阈值分类
levels = classify_tokens(importance, prefill_thresholds)

# 4. 应用稀疏化
sparse_K, sparse_V = apply_sparsity(K, V, levels)

# 5. 窗口滑动
Window_A = Window_B  # [Token_1001 ... Token_1128]
Window_B = []        # 清空

# 6. 重置累积器
attention_accumulator = zeros([B, H, 128])
```

### 🔄 第二轮及后续压缩

```python
# ===== 当前状态 =====
KV_cache = [... 稀疏化 ...] [压缩后的 873-1000] [Token_1001 ... Token_1128]
                                                ↑ Window A (上一轮的 Window B)

# ===== 生成新的 Window B =====
for k in range(128):  # 生成 Token_1129 到 Token_1256
    query = generate_query()
    keys = get_all_keys()  # 包括 Window A (Token_1001...Token_1128)
    
    attn = compute_attention(query, keys)
    attention_accumulator += attn[:, :, Window_A_indices]

# ===== Window B 满了，触发压缩 =====
# 归一化、计算 importance、分类、稀疏化、滑动
# 过程与第一轮完全相同！

# 关键：累积次数的模式仍然是 255, 254, 253, ..., 129, 128
# 因为 Window A 的 tokens 在生成时相互关注过
```

## 实现要点

### 1️⃣ 数据结构

```python
class DualWindowManager:
    def __init__(self, window_size=128):
        self.window_size = window_size
        
        # 窗口索引（逻辑划分）
        self.window_A_start = None
        self.window_A_end = None
        self.window_B_start = None
        self.window_B_end = None
        
        # Attention 累积器
        self.attention_accumulator = None  # [B, H, window_size]
        self.accumulation_count = 0  # 当前累积了多少步
        
        # Prefill 阶段的阈值
        self.prefill_thresholds = None  # (threshold_high, threshold_low)
```

### 2️⃣ 关键方法

```python
def accumulate_attention(self, attention_weights, window_A_indices):
    """
    累积 Window A 的 attention
    
    Args:
        attention_weights: [B, H, 1, seq_len] - 当前 query 的 attention
        window_A_indices: Window A 在 seq_len 中的索引
    """
    window_A_attention = attention_weights[:, :, :, window_A_indices]
    self.attention_accumulator += window_A_attention.squeeze(2)
    self.accumulation_count += 1

def normalize_attention(self):
    """
    归一化累积的 attention
    考虑不同 token 的累积次数差异
    
    Returns:
        attention_avg: [B, H, 128] - 归一化后的 attention
    """
    attention_avg = torch.zeros_like(self.attention_accumulator)
    
    for i in range(self.window_size):
        # 总累积次数 = (127 - i) + 128 = 255 - i
        total_count = 255 - i
        attention_avg[:, :, i] = self.attention_accumulator[:, :, i] / total_count
    
    return attention_avg

def compress_window_A(self, K, V, importance_calculator, sparsity_applier):
    """
    压缩 Window A
    
    Args:
        K, V: Key 和 Value tensors
        importance_calculator: 重要性计算器
        sparsity_applier: 稀疏度应用器
    """
    # 1. 归一化 attention
    attention_avg = self.normalize_attention()
    
    # 2. 计算 importance
    importance = importance_calculator.compute_diffkv_importance(attention_avg)
    
    # 3. 使用 prefill 的阈值分类
    # 注意：这里直接用 attention_avg 和阈值比较
    sparse_K, sparse_V = sparsity_applier.classify_and_apply_sparsity(
        importance, K, V, self.prefill_thresholds
    )
    
    return sparse_K, sparse_V

def slide_window(self):
    """
    窗口滑动：Window B → Window A
    """
    self.window_A_start = self.window_B_start
    self.window_A_end = self.window_B_end
    
    self.window_B_start = self.window_A_end
    self.window_B_end = None  # 待填充
    
    # 重置累积器
    self.attention_accumulator = torch.zeros_like(self.attention_accumulator)
    self.accumulation_count = 0
```

### 3️⃣ 与 MUSTAFAR 的集成

```python
# DiffSparseKV 的双窗口机制可以看作是 MUSTAFAR residual_length 的扩展

# MUSTAFAR:
residual_length = 128  # 保留最近 128 个 tokens 不压缩

# DiffSparseKV:
Window_B = 128  # 类似 MUSTAFAR 的 residual_length
Window_A = 128  # 额外的观察窗口

# 关系：
# - Window B ≈ MUSTAFAR 的 residual_length（保护最近的 tokens）
# - Window A = DiffSparseKV 的创新（等待收集足够 attention 后再压缩）
```

### 4️⃣ 简化实现（LongBench）

对于 LongBench（生成 < 128 tokens），可以简化：

```python
# 不实现双窗口，只用 residual_length
class SimplifiedDecoding:
    def __init__(self, residual_length=128):
        self.residual_length = residual_length
    
    def decode_step(self, K, V, new_K, new_V):
        # 1. 拼接新 token
        K_full = torch.cat([K, new_K], dim=2)
        V_full = torch.cat([V, new_V], dim=2)
        
        # 2. 对新 token 应用稀疏化（使用 prefill 阈值）
        # 简化：直接用 prefill 的阈值，不重新计算 importance
        
        # 3. 保持最近 residual_length 个 tokens
        if K_full.shape[2] > self.residual_length:
            # 压缩更老的 tokens
            pass
        
        return K_full, V_full
```

## 总结

### ✅ 双窗口机制的优势

1. **准确性**：每个 token 在被压缩前都被足够多的后续 tokens 观察过
2. **一致性**：累积次数的模式固定（255, 254, ..., 128），实现简单
3. **理论完整**：考虑了 causal attention 的特性，归一化合理

### 📊 累积次数总结

```
Window A 的 128 个 tokens:
- 第 0 个:   255 次 (127 + 128)
- 第 1 个:   254 次 (126 + 128)
- 第 2 个:   253 次 (125 + 128)
- ...
- 第 126 个: 129 次 (1 + 128)
- 第 127 个: 128 次 (0 + 128)

归一化：attention_avg[i] = attention_sum[i] / (255 - i)
```

### 🎯 适用场景

- **长文本生成**：生成 > 256 tokens 的任务
- **理论研究**：作为算法设计的完整方案
- **LongBench**：由于生成 < 128 tokens，可以简化实现

### 📝 实现建议

1. **完整版本**：实现双窗口机制，适用于长生成任务
2. **简化版本**：只用 residual_length，适用于 LongBench 评测
3. **文档价值**：即使不实现，也是重要的理论贡献和谈资
