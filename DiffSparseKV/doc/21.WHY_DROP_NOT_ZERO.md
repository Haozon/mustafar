# 为什么"丢弃"可以，但"置零"不行？

## 核心原因：Softmax 的数学特性

### Attention 的数学公式

```
Attention(Q, K, V) = softmax(Q·K^T / √d) · V
```

关键在于 **softmax 函数的性质**。

---

## 🔬 数学分析

### 场景设定

假设我们有 5 个 tokens，想要删除第 3 和第 5 个：

```python
Q = [q]  # 1 个 query
K = [k1, k2, k3, k4, k5]  # 5 个 keys
V = [v1, v2, v3, v4, v5]  # 5 个 values

# 我们想保留：k1, k2, k4
# 我们想删除：k3, k5
```

---

### 方法 1: 丢弃（正确）

```python
# 只保留需要的 tokens
K_compressed = [k1, k2, k4]
V_compressed = [v1, v2, v4]

# 计算 attention scores
scores = [q·k1, q·k2, q·k4] / √d
      = [s1, s2, s4]

# Softmax
weights = softmax([s1, s2, s4])
        = [exp(s1), exp(s2), exp(s4)] / (exp(s1) + exp(s2) + exp(s4))
        = [w1, w2, w4]

# 输出
output = w1·v1 + w2·v2 + w4·v4
```

**特点：**
- 分母只包含 3 项：`exp(s1) + exp(s2) + exp(s4)`
- 权重和 = 1：`w1 + w2 + w4 = 1`
- 每个权重相对较大（因为分母小）

---

### 方法 2: 置零（错误）

```python
# 保持所有 tokens，但把不需要的置零
K_zeroed = [k1, k2, 0, k4, 0]
V_zeroed = [v1, v2, 0, v4, 0]

# 计算 attention scores
scores = [q·k1, q·k2, q·0, q·k4, q·0] / √d
      = [s1, s2, 0, s4, 0]

# Softmax
weights = softmax([s1, s2, 0, s4, 0])
        = [exp(s1), exp(s2), exp(0), exp(s4), exp(0)] / (exp(s1) + exp(s2) + exp(0) + exp(s4) + exp(0))
        = [exp(s1), exp(s2), 1, exp(s4), 1] / (exp(s1) + exp(s2) + 1 + exp(s4) + 1)
        = [w1', w2', w3', w4', w5']

# 输出
output = w1'·v1 + w2'·v2 + w3'·0 + w4'·v4 + w5'·0
       = w1'·v1 + w2'·v2 + w4'·v4
```

**问题：**
- 分母包含 5 项：`exp(s1) + exp(s2) + 1 + exp(s4) + 1`
- 多了两个 `exp(0) = 1`
- 权重和仍然 = 1：`w1' + w2' + w3' + w4' + w5' = 1`
- 但 `w1' < w1`, `w2' < w2`, `w4' < w4`（因为分母变大了）
- **置零位置仍然分配了权重 w3' 和 w5'，但这些权重被浪费了！**

---

## 📊 具体数值例子

### 例子：保留 2 个 tokens，删除 3 个

```python
# 原始 scores
s1 = 2.0
s2 = 1.5
s3 = 0.1  # 要删除
s4 = 1.8
s5 = 0.2  # 要删除
```

#### 丢弃方式：

```python
scores = [2.0, 1.5, 1.8]

# Softmax
exp_scores = [exp(2.0), exp(1.5), exp(1.8)]
           = [7.39, 4.48, 6.05]

denominator = 7.39 + 4.48 + 6.05 = 17.92

weights = [7.39/17.92, 4.48/17.92, 6.05/17.92]
        = [0.412, 0.250, 0.338]

# 权重和 = 1.000 ✅
# 每个有效 token 得到合理的权重
```

#### 置零方式：

```python
scores = [2.0, 1.5, 0.0, 1.8, 0.0]

# Softmax
exp_scores = [exp(2.0), exp(1.5), exp(0.0), exp(1.8), exp(0.0)]
           = [7.39, 4.48, 1.00, 6.05, 1.00]

denominator = 7.39 + 4.48 + 1.00 + 6.05 + 1.00 = 19.92

weights = [7.39/19.92, 4.48/19.92, 1.00/19.92, 6.05/19.92, 1.00/19.92]
        = [0.371, 0.225, 0.050, 0.304, 0.050]
          ↑       ↑       ↑       ↑       ↑
        有效    有效    浪费    有效    浪费

# 权重和 = 1.000
# 但有 0.050 + 0.050 = 0.100 (10%) 的权重被浪费在置零位置！
# 有效 token 的权重被稀释了
```

**对比：**

| Token | 丢弃方式 | 置零方式 | 差异 |
|-------|---------|---------|------|
| k1 (s=2.0) | 0.412 | 0.371 | -10% |
| k2 (s=1.5) | 0.250 | 0.225 | -10% |
| k3 (删除) | - | 0.050 | 浪费 |
| k4 (s=1.8) | 0.338 | 0.304 | -10% |
| k5 (删除) | - | 0.050 | 浪费 |

**结论：置零方式让每个有效 token 的权重减少了约 10%！**

---

## 🎯 为什么这会导致模型效果差？

### 1. 信息被稀释

```python
# 丢弃方式的输出
output = 0.412·v1 + 0.250·v2 + 0.338·v4

# 置零方式的输出
output = 0.371·v1 + 0.225·v2 + 0.304·v4
#        ↓ 减少了 10%

# 每个有效 token 的贡献都减少了
# 总信息量减少了 10%
```

### 2. 累积效应

```python
# 如果删除 35% 的 tokens（你的配置）
# 置零方式会浪费约 35% 的 attention 权重
# 有效 token 的权重只有原来的 65%

# 在多层 Transformer 中：
# Layer 1: 信息保留 65%
# Layer 2: 信息保留 65% × 65% = 42%
# Layer 3: 信息保留 65% × 65% × 65% = 27%
# ...
# 32 层后：信息几乎完全丢失！
```

### 3. 梯度和训练的影响

即使在推理阶段：
```python
# 置零的 K 和 V 仍然占用内存
# 仍然参与计算（虽然贡献为 0）
# 但它们的存在改变了 softmax 的归一化

# 这相当于在 attention 分布上加了一个"噪声"
# 这个噪声会破坏模型学习到的 attention 模式
```

## 🔬 实验验证

### 简单的 Python 验证

```python
import torch
import torch.nn.functional as F

# 设置
Q = torch.randn(1, 1, 1, 128)
K = torch.randn(1, 1, 1000, 128)
V = torch.randn(1, 1, 1000, 128)

# 选择要保留的 tokens（前 650 个）
keep_indices = torch.arange(650)

# 方法 1: 丢弃
K_drop = K[:, :, keep_indices, :]  # [1, 1, 650, 128]
V_drop = V[:, :, keep_indices, :]
scores_drop = (Q @ K_drop.transpose(-2, -1)) / (128 ** 0.5)
attn_drop = F.softmax(scores_drop, dim=-1)
output_drop = attn_drop @ V_drop

# 方法 2: 置零
K_zero = K.clone()
V_zero = V.clone()
K_zero[:, :, 650:, :] = 0
V_zero[:, :, 650:, :] = 0
scores_zero = (Q @ K_zero.transpose(-2, -1)) / (128 ** 0.5)
attn_zero = F.softmax(scores_zero, dim=-1)
output_zero = attn_zero @ V_zero

# 对比
print("丢弃方式的 attention 权重和:", attn_drop.sum().item())  # 1.0
print("置零方式的 attention 权重和:", attn_zero.sum().item())  # 1.0

print("\n有效位置的平均权重:")
print("丢弃方式:", attn_drop[:, :, :, :650].mean().item())  # 约 1/650
print("置零方式:", attn_zero[:, :, :, :650].mean().item())  # 约 0.65/1000

print("\n置零位置的权重:")
print("置零方式:", attn_zero[:, :, :, 650:].sum().item())  # 约 0.35（浪费！）

print("\n输出差异:")
print("L2 distance:", torch.norm(output_drop - output_zero[:, :, :, :]).item())
```

**预期结果：**
```
丢弃方式的 attention 权重和: 1.0
置零方式的 attention 权重和: 1.0

有效位置的平均权重:
丢弃方式: 0.001538  (1/650)
置零方式: 0.000650  (0.65/1000)

置零位置的权重:
置零方式: 0.350  (35% 的权重被浪费！)

输出差异:
L2 distance: 2.456  (输出有显著差异)
```

## 💡 类比理解

### 类比 1: 分蛋糕

**丢弃方式：**
- 有 10 个人，但只有 6 个人来了
- 把蛋糕分成 6 份，每人 1/6
- 每个人都吃饱了 ✅

**置零方式：**
- 有 10 个人，但只有 6 个人来了
- 仍然把蛋糕分成 10 份，每人 1/10
- 4 份给了空位（浪费）
- 6 个人每人只得到 1/10，没吃饱 ❌

### 类比 2: 投票

**丢弃方式：**
- 有 100 张选票，65 张有效，35 张作废
- 只统计 65 张有效票
- 每张有效票的权重 = 1/65 = 1.54%

**置零方式：**
- 有 100 张选票，65 张有效，35 张作废
- 统计所有 100 张票（包括作废票）
- 每张有效票的权重 = 1/100 = 1%
- 作废票仍然占用了 35% 的"投票权"

## 🎯 总结

### 为什么"丢弃"可以：

1. ✅ **Softmax 归一化正确**：分母只包含有效 tokens
2. ✅ **权重分配合理**：每个有效 token 得到应有的权重
3. ✅ **信息不被稀释**：100% 的 attention 权重用于有效 tokens
4. ✅ **节省计算和内存**：真正减少了序列长度

### 为什么"置零"不行：

1. ❌ **Softmax 归一化错误**：分母包含置零位置的 exp(0) = 1
2. ❌ **权重被浪费**：置零位置仍然分配了权重（但贡献为 0）
3. ❌ **信息被稀释**：有效 tokens 的权重被稀释了
4. ❌ **不节省计算**：仍然计算所有位置的 attention
5. ❌ **累积效应**：多层后信息损失严重

### 数学本质：

```
丢弃：softmax([s1, s2, s4]) → 权重和 = 1，分配给 3 个 tokens
置零：softmax([s1, s2, 0, s4, 0]) → 权重和 = 1，分配给 5 个位置（2 个浪费）

exp(0) = 1 ≠ 0  ← 这是关键！
```

---

## 🔧 如何修复

你需要把代码从"置零"改为"丢弃"：

```python
# 当前（置零）
pruned_keys[level_mask] = 0.0  # ❌

# 应该改为（丢弃）
keep_mask = (sparsity_levels != 2)
keep_indices = torch.where(keep_mask[0, 0, :])[0]
pruned_keys = key_states[:, :, keep_indices, :]  # ✅
```

这就是为什么你的模型效果差的根本原因！
